### Rule 1: Say No to Spaces

While spaces in file and directory names might read nicer for regular users, they can cause serious problems, especially when working with command line or terminal commands.

==Why Spaces Break Things==

In command-line tools, a space often signifies moving on to a new part of the command, not that it is part of the file name. When using files with spaces, you are forced to add additional characters, called escape characters, or wrap the names in quotation marks or other symbols.

Spaces are also problematic in web development, as they are not valid in URLs and must be replaced byÂ `%20`Â characters. At best, dealing with spaces is annoying, and at worst, they can break your code or commands.

Professional Separators

When dealing with names that contain multiple words, professionals typically use one of these four options instead of spaces:

2.Â **Hyphens (-):**Â These are very popular in web files, though some tools or programming languages may treat hyphens as subtraction operators (e.g.,Â `my-project-file.py`).

3.Â **CamelCase:**Â The first character is lowercase, and then each additional word uses a capital letter (e.g.,Â `myProjectFile.py`).

4.Â **PascalCase (or Proper Case/UpperCamelCase):**Â Each word starts with a capital letter, including the very first character (e.g.,Â `MyProjectFile.py`).

### Rule 2: Avoid Risky Special Characters

Generally, the safest characters to use in file and directory names are lettersÂ **A to Z**, numbersÂ **0 to 9**, and theÂ **underscore**. While hyphens are usually safe, you should avoid special characters like dollar signs ($), exclamation points (!), and similar symbols.

Using these special characters is risky because depending on the software or tools you are using, they may cause problems. You should avoid special characters at all costs, except in rare cases where an application specifically requires them.

The Dot Character

The dot character (`.`) is a special case. It is used for file extensions and for denoting hidden files. You can, and must, use it for these specific purposes in file and directory names, but otherwise, you should avoid using it.

### Rule 3: Be Descriptively Concise

There is an art to naming things, and it is a surprisingly difficult aspect of being a technology professional. The goal is to select names that areÂ **descriptive enough to be clear to others**Â but are not so long that they become a pain to type.

If you master this balance, your future self and your co-workers will greatly appreciate clear, descriptive names that explain the purpose of the file or directory.

As you move into professional technology environments, you quickly learn that letter casing often matters. For instance, capitalÂ `Hell`Â is usually not the same as lowercaseÂ `hello`. The exact behaviour depends on the tool and software you are using.

### Rule 5: Use YYYYMMDD for Sorted Dates

The Sorting Problem

If you use regional date formats (likeÂ `MM-DD-YYYY`), the files will not be sorted chronologically. This is because sorting works by comparing each character from left to right. For example, if you sort files where the month is listed first, the file starting with "12" will appear before the file starting with "2" (e.g.,Â `12_report`Â will sort beforeÂ `2_report`), meaning the files will be out of order.

The Professional Solution

To ensure proper sorting in ascending order, you should always use theÂ **year month day (YYYYMMDD)**Â format for dates in file and directory names.

This format, which is the international standard, involves using leading zeros for the month and day. When files are named usingÂ `YYYYMMDD`, the names will be consistent, they will always be the same length, and each character can be compared properly, ensuring correct chronological sorting.

### Bonus Rule: Be Consistent

Once you establish a naming pattern within a project, you mustÂ **stick with it**.
